## Phase 1: 調査（search-first）

1. タスクに関連する既存コードを検索する
2. 既存パターン（命名規則、ディレクトリ構造、エラーハンドリング）を把握
3. package.json の依存関係を確認
4. `.claude/rules/` に関連するルールがないか確認
5. `.rwl/lessons/` に関連する失敗/成功パターンがないか確認
6. 同名・類似機能の実装が既にないか確認する

## Phase 2: 変更対象の限定

- 変更するファイル一覧（パスを列挙）
- 変更しないファイル一覧（スコープ外を明示）
- 新規作成するファイル（あれば）

## Phase 3: DoD の明文化

- [ ] 具体的な完了条件
- [ ] テスト: `<テストコマンド>` がパスする
- [ ] 既存テストが壊れていない

## Phase 4: ユーザー確認

Phase 1-3の結果をまとめて報告し、承認を得る。承認なしに実装に進まない。

## Phase 5: 実装

Phase 2の対象ファイルのみ変更し、Phase 3のDoDを全て満たすまで実装する。

## Phase 6: 品質ゲート

実装完了後、`/quality-gate` を実行して全チェックをパスすること。
GATE FAILED の場合は修正してから再実行する。

## 禁止事項

- Phase 1-3をスキップしていきなり実装に入る
- Phase 2で限定した対象外のファイルを変更する
- DoDに書いていない変更を追加する
- 同名・類似機能の実装があるのに抽象ユーティリティを新設する
